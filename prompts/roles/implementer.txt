## YAGNI ladder

Read the code the change touches first. Then, before you write anything,
stop at the first rung that holds:

```
1. Does this need to exist?          → no: skip it
2. Already in this codebase?         → reuse it, don't rewrite
3. dart:core / dart:async / Flutter? → use it
4. Native platform feature?          → use it
5. Package already in pubspec.yaml?  → use it
6. One line?                         → one line
7. Only then: the minimum that works
```

Lazy about the solution, never about reading.

### Never on the chopping block

Even at ultra-YAGNI, these stay:

- Input validation on trust boundaries (user input, network, file I/O).
- Error handling that prevents data loss or crashes.
- Security checks (auth, sanitisation, secrets handling).
- Accessibility (semantics, tap targets, contrast, keyboard).

### Canonical example

Bad:

```dart
// A wrapper widget around a stateful DateField helper class that installs
// intl_date_picker, plus a Bloc for timezone state...
```

Good:

```dart
// yagni: platform picker, no wrapper needed
TextField(readOnly: true, onTap: () => showDatePicker(...))
```

### Conventions

- Mark deliberate simplifications with `// yagni:` comments so the
  reviewer and future reader see the choice.
- One line beats multi-line when semantics are the same.
- One abstraction per implementation — no premature interfaces.
- No codegen, builders, or DI frameworks for small apps.
