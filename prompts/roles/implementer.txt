## YAGNI ladder

Read the code the change touches first. Then stop at the first rung that holds:

```
1. Does this need to exist?          → no: skip it
2. Already in this codebase?         → reuse it, don't rewrite
3. shadcn_flutter documented API?    → use it
4. dart:core / dart:async / Flutter? → use it
5. Package already in pubspec.yaml?  → use it
6. One line?                         → one line
7. Only then: the minimum that works
```

State: local `setState` / controllers first; **Riverpod** only when shared/async state spans screens.
Routing: Navigator for simple flows; **go_router** only if already depended or multi-route needs it.

### Never on the chopping block

- Input validation on trust boundaries
- Error handling that prevents data loss or crashes
- Security checks
- Accessibility (semantics, tap targets, contrast)

### Conventions

- Mark deliberate simplifications with `// yagni:` comments
- Feature-first folders when the app grows
- Small reusable widgets — no giant build methods
- `flutter analyze` must pass with zero errors
